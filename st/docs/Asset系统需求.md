# 当前 Asset 系统规划总结

## 1. 核心定位

Asset 不再是“角色升级点”，而是：

```text
团队卡组构筑资源
```

也就是说，Asset 升级的不是某个角色，而是整个队伍/当前周目的“激活卡组”。

无论当前上场的是谁，卡组都存在。
卡牌根据自己的生效条件，自动作用于“有资格的人”。

例如：

```text
Mana Max +5
只对有 Mana User 标签的角色生效
不会作用于 KAZU
```

这样可以避免：

```text
KAZU 学魔法 OOC
RINO 满级大佬还要升级的违和感
临时队友/敌人无法长期培养的问题
```

# 2. 卡槽系统

## 2.1 卡槽结构

初始：

```text
通用卡槽：4
Void 专属卡槽：2
```

上限：

```text
通用卡槽：8
Void 专属卡槽：2
```

最终结构：

```text
8 + 2
```

---

## 2.2 通用卡槽规则

通用卡槽可以放：

```text
通用卡
德州卡
小游戏卡
Moirai / Chaos / Psyche 卡
Void 卡
指定角色卡
```

也就是说：

```text
通用槽什么都能放
```

---

## 2.3 Void 专属槽规则

Void 专属槽只能放：

```text
Void 系卡
KAZU 专属卡
```

不能放：

```text
Moirai 卡
Chaos 卡
Psyche 卡
Mana User 卡
德州通用魔法卡
```

设计目的：

```text
给 KAZU 保底收益
避免“奖励全是魔法，男主吃不到”的心理落差
```

---

# 3. 扩容规则

通用卡槽从 4 个扩到 8 个。

扩容价格：

```text
第 5 槽：1 Asset
第 6 槽：1 Asset
第 7 槽：2 Asset
第 8 槽：2 Asset
```

总扩容成本：

```text
6 Asset
```

Void 专属槽不扩容，固定 2 个。

---

# 4. Asset 使用方式

每次结算获得 Asset 后，玩家二选一：

```text
1. 花 Asset 扩容卡槽
2. 花 Asset 抽卡池
```

不做复杂角色升级树。

目前四种：

```text

```

大致含义：

```text

```

# 7. 卡牌存放规则

## 7.1 无仓库

不采用仓库/背包模式。

也就是说：

```text
抽到并选择的卡，必须立刻进入激活区
```

没有：

```text
先放仓库
之后再装配
临时保存
```

---

## 7.2 满槽替换

如果卡槽满了，玩家选择新卡时，必须：

```text
选择一张旧卡替换
```

被替换的卡：

```text
直接销毁
```

不返还、不进仓库。

---

## 7.3 Void 卡的放置

Void 卡可以放入：

```text
通用卡槽
Void 专属槽
```

非 Void 卡不能放入 Void 专属槽。

---

# 8. 卡牌生效对象

卡牌不绑定“升级某个角色”，而是通过标签判断生效对象。

目标类型包括：

```text
通用 / 全员
Mana User
Void
Moirai
Chaos
Psyche
指定角色，例如 RINO / KAZU / SIA
```

示例：

```text
Mana Max +8
目标：Mana User
效果：所有有 Mana 功能的角色 Mana 上限 +8
KAZU 不生效
```

```text
Void 绝缘强化
目标：Void
效果：只对 KAZU 生效
```

```text
Moirai 攻击 +12%
目标：Moirai
效果：只强化 Moirai 系角色/技能
```

# 10. 卡牌按功能分类

持续若干节点。


通用升级卡属于消耗品。

购买后立刻选择目标技能升级，之后卡牌销毁，不占槽。

# 11. Mana 消耗规则

Mana 减费分两类：

```text
固定减费
百分比减费
```

结算顺序：

```text
先减固定值
再结算百分比减费
```

百分比减费上限：

```text
最高 -66%
```

固定减费：

```text
无限制
```

最终 Mana 消耗：

```text
不能为负
最低为 1
```

但如果固定减费把小技能砍到 0，视为白嫖：

```text
原始消耗 - 固定减费 <= 0
最终消耗 = 0
```

更准确的逻辑可以写成：

```text
剩余消耗 = 原始 Mana - 固定减费

如果 剩余消耗 <= 0：
    最终消耗 = 0
否则：
    最终消耗 = max(1, ceil(剩余消耗 × (1 - min(百分比减费, 66%))))
```

这样：

```text
小技能可能被固定减费砍成 0
大技能通常不会
百分比减费不能把技能压成负数
不会电表倒转
```

# 12. 增幅规则

技能效果增幅不设上限。

也就是说：

```text

```

可以叠加，没有 66% 上限。

# 14. 德州技能卡

## 14.1 技能卡通用规则

技能卡不能叠加。

每个技能只能存在一张。

规则：

```text
如果已有低级卡，获得高级卡时，直接替换为高级卡
如果已有同级卡，再获得同级卡时，直接合成下一级
如果已经满级，不再刷出该技能卡
如果已有更高级卡，不再刷出低级卡
```

玩家不需要手动处理这些替换。

# 20. 当前系统优点

这套 Asset 系统目前解决了几个关键问题：

## 20.1 不再需要全角色养成界面

不需要给每个角色做技能树。

只需要维护：

```text
团队卡组
卡槽
卡池
卡牌效果
```

---

## 20.2 避免叙事 OOC

```text
KAZU 不会学魔法
RINO 不需要从低级法术练起
临时队友不需要培养
敌人也不需要升级
```

卡牌只是团队层面的机制强化。

---

## 20.3 代码结构清晰

主要数据结构可以是：

```text
world.act.reserve.asset
general_slots_unlocked
void_slots_unlocked = 2
active_general_cards[]
active_void_cards[]
card_pool[]
owned_skill_levels{}
```

核心流程：

```text
结算获得 Asset
选择扩容或抽卡
抽卡生成 3 张候选
选择 1 张
判断是否升级/合成/占槽
如果占槽且满了，则替换旧卡
保存当前激活卡组
进入下一段
```

---

# 21. 当前仍需之后确认的小点

后面还需要定几个细节：

```text
1. 每次抽卡是展示 3 张还是 4 张
2. 替换旧卡时是否允许取消购买
3. 消耗品升级卡是否也算一次抽卡结果
4. 同名数值卡是否无限叠加，还是设叠加上限
5. 彩卡是否全部唯一
6. Void 专属槽满了时，新 Void 卡是否优先替换 Void 槽，还是允许放通用槽
7. 神卡是否允许被替换销毁
8. 高风险高收益卡是否可叠加
```

但基础框架已经明确了。

---

# 一句话总结

当前 Asset 系统是：

```text
一个团队层面的肉鸽卡组系统。
```

玩家用 Asset 在结算阶段选择：

```text
扩容卡槽
或
抽取不同等级卡池
```

卡牌进入激活卡组后，通过标签判断生效对象。
初始 4 个通用槽 + 2 个 Void 槽，最终 8 个通用槽 + 2 个 Void 槽。
没有仓库，满槽必须替换，旧卡销毁。
这样既保留构筑乐趣，也避免复杂角色养成和设定冲突。
