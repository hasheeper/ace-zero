# Mahjong Future-Proofing 参考

本文用于记录 `majiang` 项目在当前四麻主线开发中，哪些地方**必须解耦、不能写死**，以便未来兼容：

- 三麻
- 异能 / 魔运
- 更复杂的 AI
- 异能 UI 与特殊演出

本文不是功能清单，而是**工程边界与约束清单**。

核心原则只有一句话：

**现在可以先不实现，但不能把以后要走的路写死。**

---

## 1. 当前决策总览

### 1.1 当前主线

当前主线仍然是：

- 标准四麻
- 基础单局闭环
- 后续扩展到跨局
- 当前先跑基础 AI

### 1.2 当前不立即实现的内容

当前暂不立即实现：

- 三麻完整规则
- 三麻专属 UI
- 复杂异能系统
- 复杂魔运系统
- 高级 AI
- 大量异能演出

### 1.3 但必须提前兼容的内容

虽然现在不做，但以下方向必须提前为接口留口：

- 三麻 ruleset
- 异能 / 魔运对牌墙与信息层的影响
- 更复杂 AI 的输入输出协议
- 异能 UI 的动作与事件流

结论：

**现在允许“功能缺失”，但不允许“结构锁死”。**

---

## 2. 什么叫“不能写死”

这里的“不能写死”，不是要求现在就做一个万能插件系统。

更准确地说，是：

- 不能把关键规则散写在 UI 里
- 不能把未来一定会变化的结构当常量写死
- 不能让 AI、runtime、牌墙、UI 彼此互相直接耦死
- 不能让未来的扩展必须推翻当前主流程

允许的简化：

- 当前只跑四麻
- 当前只跑无异能
- 当前只跑 simple AI
- 当前 UI 只显示基础动作

不允许的简化：

- 永远假定只有 4 家
- 永远假定只有随机发牌
- 永远假定 AI 看到的就是真相
- 永远假定动作只有打牌 / 立直
- 永远假定前端事件只有摸牌 / 出牌 / 和牌

---

## 3. 必须提前解耦的模块总表

以下几类必须提前解耦：

1. 座位与人数
2. 规则集
3. 牌墙与发牌
4. 主流程状态机
5. 动作系统
6. 信息视图
7. AI 输入输出
8. AI 性格与难度
9. 技能 / 异能 / 反异能接口
10. UI 事件流与特效触发
11. 日志与回放

下面逐项展开。

---

## 4. 座位与人数：不能写死四家

### 4.1 为什么不能写死

如果把以下内容写死：

- 永远 `bottom/right/top/left`
- 永远 4 家轮转
- 永远有 top seat
- 永远 4 个分数位

那么以后做三麻时，会在这些层面全线返工：

- runtime
- UI
- 动作响应
- AI 轮转逻辑

### 4.2 当前必须抽象的内容

即使当前只跑四麻，也必须有：

- `tableSize`
- `activeSeats`
- `seatOrder`
- `hiddenSeats`
- `nextSeat(seatKey)`

### 4.3 当前可以先不做的内容

可以暂时不做：

- 三麻牌库
- 三麻规则
- 三麻桌型视觉
- 三麻专属 AI

结论：

**现在不做三麻功能，但必须保证人数与座位层不是四人硬编码。**

---

## 5. 规则集：不能把四麻规则散写进 runtime

### 5.1 为什么不能写死

如果把这些内容散写在 runtime / UI / action panel 里：

- 总牌数 136
- 永远能吃
- 永远起手 13 张
- 永远死牌区固定四麻逻辑

以后做三麻、魔改规则、特殊模式时会非常痛苦。

### 5.2 必须解耦成 ruleset profile

规则集至少要能表达：

- `seatCount`
- `tileSet`
- `handSize`
- `deadWallSize`
- `enableChi`
- `enablePeng`
- `enableGang`
- `enableRiichi`
- `enableDora`
- `enableSpecialActions`

### 5.3 当前建议

当前只正式支持：

- `riichi-4p`

但结构必须允许未来再加入：

- `riichi-3p-sanma`

结论：

**当前可以只有一个 ruleset，但不能没有 ruleset 概念。**

---

## 6. 牌墙 / 发牌 / 魔运：不能写死“纯随机”

### 6.1 为什么不能写死

如果发牌和摸牌直接散落在 runtime 里，且默认就是：

- 纯随机开局
- 纯随机摸牌
- 没有策略层

那么以后这些都很难接：

- 发好牌
- 发烂牌
- 开局偏置
- 中盘偏置
- 反异能抵消发牌影响

### 6.2 必须解耦的层

牌墙与策略必须分层：

- `majiang-core`
- `wall-service`
- `draw-policy`

推荐职责：

- `majiang-core`
  - 规则底层
  - 真始牌山能力

- `wall-service`
  - 拥有牌墙
  - 发牌 / 摸牌 / 杠摸 / 翻宝牌
  - 输出可追踪元信息

- `draw-policy`
  - 决定是否偏置
  - 决定发牌 / 摸牌建议
  - 不直接拥有规则真相

### 6.3 当前必须保留的接口

至少要有：

- `chooseInitialHands`
- `chooseDraw`
- `chooseGangDraw`
- `revealDora`

### 6.4 当前默认实现

当前默认可以是：

- no-op policy
- 纯真实随机牌山

结论：

**现在可以不做魔运，但不能把“随机发牌”写死成唯一来源。**

---

## 7. 主流程状态机：不能写死成“一条摸打线”

### 7.1 为什么不能写死

如果 runtime 只按“当前玩家摸牌 -> 打牌 -> 下家摸牌”直线推进，会很难兼容：

- 吃碰杠响应
- 荣和响应
- 技能抢响应
- 反异能介入
- 多段动作窗口

### 7.2 必须显式的状态

至少要有这些状态概念：

- `await_draw`
- `await_discard`
- `await_reaction`
- `await_resolution`
- `round_end`

### 7.3 当前必须避免的写法

要避免：

- UI 点击直接推动真相状态
- 某个按钮自己改完整桌面状态
- 动作与动画耦死

结论：

**即使当前只有基础动作，也必须按“状态机”而不是“页面脚本串流程”来写。**

---

## 8. 动作系统：不能把按钮和动作种类写死

### 8.1 为什么不能写死

如果动作面板只支持固定按钮：

- 打牌
- 立直
- 自摸

以后很难接：

- 吃 / 碰 / 杠
- 九种九牌
- 技能动作
- 反制动作
- 异能确认动作

### 8.2 正确方式

runtime 应输出结构化动作列表：

```js
{
  type: 'discard',
  key: 'discard:m5',
  label: '打出五万',
  group: 'basic',
  priority: 10,
  payload: {}
}
```

以后可扩展成：

- `chi`
- `peng`
- `gang`
- `riichi`
- `hule`
- `skill`
- `counter-skill`

### 8.3 当前建议

前端动作面板只负责：

- 渲染动作
- 按 `group / priority` 排列
- 将动作回传 runtime

不负责：

- 自己判断动作是否合法
- 自己写异能特殊逻辑

结论：

**动作系统必须数据驱动，不能把当前按钮集合写死成未来唯一动作集合。**

---

## 9. 信息视图：不能默认 AI 看到的就是真相

### 9.1 为什么不能写死

异能与 AI 扩展里，最容易踩坑的就是信息层。

未来可能出现：

- AI 只能看到公开信息
- 某技能让 AI 额外看到部分危险牌信息
- 某异能给出“模糊透视”
- 某反异能会干扰透视结果

如果 AI 直接绑定完整真相状态，以后非常难改。

### 9.2 必须分层的 view

至少要区分：

- `truthState`
- `playerView`
- `aiView`

推荐解释：

- `truthState`
  - runtime 真相

- `playerView`
  - 正常玩家可见信息

- `aiView`
  - AI 决策时看到的视图
  - 可以被透视、反透视、技能、难度修正

### 9.3 当前建议

即使现在 simple AI 先用接近真相的简化输入，也应通过：

- `buildAiView(truthState, seat, options)`

而不是直接把 runtime 内部对象塞给 AI。

结论：

**信息视图必须先抽象，否则透视类异能和复杂 AI 会极难接。**

---

## 10. AI 输入输出：不能只支持“打一张牌”

### 10.1 为什么不能写死

如果 AI 接口只支持：

- 输入一份桌面状态
- 输出一张要打的牌

那它后面几乎无法优雅接：

- 吃碰杠
- 荣和 / 自摸
- 立直
- 技能发动
- 技能反制

### 10.2 必须统一成动作协议

AI 输出必须是动作对象，而不是裸值：

```js
{
  type: 'discard',
  key: 'discard:p5',
  payload: {
    tile: 'p5'
  },
  meta: {}
}
```

这样未来才能统一扩展：

- `discard`
- `meld`
- `riichi`
- `hule`
- `skill`
- `counter-skill`

### 10.3 AI 输入也不能写死

AI 输入应包含：

- 当前动作窗口
- 当前可行动作列表
- 当前视图
- 当前 round / match context
- 当前 profile / difficulty

而不是只给：

- 手牌字符串

结论：

**AI 输入输出协议必须先统一，否则以后复杂 AI 与技能 AI 会互相打架。**

---

## 11. AI 性格与难度：不能把“风格”写进算法本体

### 11.1 为什么不能写死

如果简单 AI、普通 AI、困难 AI 分别写成三套完全不同逻辑，
或者把“激进型”“稳健型”直接散写在主算法里，后面会非常难维护。

### 11.2 推荐结构

推荐拆成：

- `base evaluator`
- `difficulty policy`
- `profile`

解释：

- `base evaluator`
  - 负责基础数学评估

- `difficulty policy`
  - 控制搜索深度
  - 控制是否启用危险度
  - 控制是否启用 rollout

- `profile`
  - 调整偏好
  - 不破坏数学底线

### 11.3 当前必须避免的错误

不要写成：

- 困难 AI 是一套独立黑盒
- 激进型 AI 是另一套完全独立脚本

正确目标是：

**同一套底层评估器，不同难度和性格只是不同策略配置。**

结论：

**AI 性格、AI 难度、AI 异能能力必须分层，不能糊成一个“大脑”。**

---

## 12. 技能 / 异能 / 反异能：不能直接塞进 UI 或 core

### 12.1 为什么不能写死

未来异能很可能影响：

- 发牌
- 摸牌
- 视图
- 动作窗口
- 动作优先级
- AI 决策

如果直接把技能逻辑写进：

- `majiang-core`
- 前端按钮点击事件
- 某个 AI 分支

会快速失控。

### 12.2 推荐解耦位置

技能 / 异能 / 反异能应主要挂在：

- `wall-service`
- `runtime`
- `engine/ai`
- `view builder`

不应挂在：

- `majiang-core`
- DOM 事件处理器

### 12.3 当前至少要预留的钩子

至少要有：

- `beforeRoundStart`
- `beforeDraw`
- `afterDraw`
- `beforeAction`
- `afterAction`
- `buildAiView`
- `evaluateSkillUse`
- `beforeResolveReaction`
- `afterResolveReaction`

结论：

**异能系统不必现在实现，但必须有合理挂载点，不能未来只能靠硬插 if/else。**

---

## 13. UI 事件流与异能演出：不能把动画种类写死

### 13.1 为什么不能写死

如果前端只认识：

- 摸牌动画
- 出牌动画
- 和牌动画

未来异能 UI 会很难接，因为技能表现可能包括：

- 目标锁定
- 牌墙波动
- 透视提示
- 反制闪断
- 视野污染
- 角色立绘触发

### 13.2 正确方式

前端应基于事件流驱动，而不是只认少量固定动作。

建议事件种类可扩展，例如：

- `tile-drawn`
- `tile-discarded`
- `meld-called`
- `riichi-declared`
- `round-ended`
- `skill-triggered`
- `skill-countered`
- `perception-updated`
- `wall-biased`
- `status-applied`

### 13.3 当前建议

现在即使没有异能演出，也应让前端渲染层具备：

- 接收通用事件
- 按事件类型分派动画
- 未识别事件可静默跳过

结论：

**异能 UI 现在可以先不做，但前端事件系统不能写死成只认识传统麻将动作。**

---

## 14. 日志与回放：不能只记结果

### 14.1 为什么不能写死

如果日志只记录：

- 谁摸了什么
- 谁打了什么

那以后排查这些问题会非常痛苦：

- 这张牌是不是被魔运偏置过
- AI 为什么在这里用了技能
- 这次透视结果为什么不一样
- 反异能为何生效 / 未生效

### 14.2 必须能记录的元信息

建议日志至少支持：

- 事件类型
- 作用对象
- 动作来源
- 是否有技能影响
- 是否有反制影响
- AI profile / difficulty
- evaluator 关键理由

### 14.3 当前建议

即使当前还是基础日志，也要尽量用结构化 entry：

```js
{
  scope: 'runtime',
  type: 'tile-discarded',
  actor: 'bottom',
  tile: 'm5',
  meta: {}
}
```

结论：

**日志和回放必须面向未来扩展设计，不然 AI 和异能 bug 将几乎不可排查。**

---

## 15. 当前阶段“必须解耦”和“可以先简单”的边界

### 15.1 当前必须解耦

- seat topology
- ruleset
- wall-service / draw-policy
- runtime 状态机
- 动作系统
- truth/player/ai view
- AI 输入输出协议
- AI difficulty / profile / evaluator 分层
- 技能钩子
- UI 事件流
- 结构化日志

### 15.2 当前可以先简单

- 只支持 `riichi-4p`
- 只支持无异能默认流程
- 只跑 simple AI
- UI 先不做三麻
- UI 先不做复杂异能演出
- draw policy 默认 no-op
- AI view 暂时可接近公开信息基础版

### 15.3 当前不该做的过度设计

不要现在就做：

- 完整插件市场式技能系统
- 超通用脚本化事件 DSL
- 过深的 runtime module 嵌套
- 完整通用三麻 / 两麻 / 魔改麻将统一系统

当前目标不是万能引擎，而是：

**在四麻主线可推进的前提下，不制造未来扩展的结构性障碍。**

---

## 16. 推荐的工程原则

推荐坚持以下原则：

1. 默认实现可以简单，但接口不能僵死
2. 真相状态、AI 视图、前端展示必须分开
3. 规则、牌墙、AI、UI 不能互相越界
4. 异能和魔运不要污染 `majiang-core`
5. 性格与难度不要重写 AI，只修正偏好和能力开关
6. 三麻现在不做，但四麻代码不能阻断三麻
7. UI 现在不做异能大演出，但事件系统要能接

---

## 17. 最终结论

### 17.1 关于三麻

当前结论：

- 现在不做三麻功能
- 但必须为三麻保留接口

### 17.2 关于异能 / 魔运

当前结论：

- 现在可以默认无异能
- 但不能把牌墙、信息层、动作入口写死

### 17.3 关于复杂 AI

当前结论：

- 现在可以先只跑 simple AI
- 但 AI 输入输出、视图、动作协议、profile 分层必须先留好

### 17.4 关于异能 UI

当前结论：

- 现在可以不做复杂异能演出
- 但 UI 事件流与动作系统必须可扩展

### 17.5 一句话总结

当前四麻主线开发应遵循的核心原则是：

**先做简单实现，但必须在 seat、ruleset、wall、runtime、view、AI、action、UI event 这些关键层面保持解耦，避免把未来的三麻、异能、复杂 AI、异能 UI 写死在今天的基础实现里。**
